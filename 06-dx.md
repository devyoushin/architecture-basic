# 06. Direct Connect (DX) 전략

## 개요

Direct Connect는 온프레미스 데이터센터와 AWS를 전용 사설 회선으로 연결하는 서비스입니다.
공용 인터넷을 경유하지 않아 안정적인 대역폭과 낮은 지연시간을 제공합니다.

---

## Direct Connect vs VPN

| 항목 | Direct Connect | Site-to-Site VPN |
|------|---------------|-----------------|
| 연결 방식 | 전용 사설 회선 | 인터넷 경유 암호화 터널 |
| 대역폭 | 1Gbps ~ 100Gbps | 최대 1.25Gbps |
| 지연시간 | 낮고 안정적 | 인터넷 상태에 따라 변동 |
| 가용성 | 물리 회선 의존 | 인터넷 이중화로 높음 |
| 비용 | 높음 (포트 + 회선 비용) | 낮음 |
| 암호화 | 기본 미제공 (MACsec 선택) | IPsec 암호화 기본 |
| 구축 기간 | 수 주 ~ 수 개월 | 즉시 가능 |

> **권장 패턴:** DX를 주 연결로, VPN을 백업(Failover) 연결로 구성

---

## Direct Connect 연결 구조

### 전체 연결 구성도

```
온프레미스 데이터센터
        │
   고객 라우터 (CE Router)
        │ 전용회선
   DX Location (코로케이션)
        │
   AWS DX Router
        │
   Direct Connect Gateway (DXGW)
        │
   Transit Gateway (TGW)     ← Network Account
        │
   각 계정 VPC
```

### 핵심 구성 요소

| 구성 요소 | 역할 |
|----------|------|
| DX Connection | 물리 전용 회선 (1G, 10G, 100G) |
| Virtual Interface (VIF) | 논리 연결 (Private VIF / Transit VIF / Public VIF) |
| Direct Connect Gateway (DXGW) | 멀티 리전, 멀티 TGW 연결을 위한 글로벌 게이트웨이 |
| Transit Gateway (TGW) | VPC 연결 허브 (DXGW와 연동) |

---

## Virtual Interface (VIF) 종류

| VIF 타입 | 연결 대상 | 용도 |
|----------|---------|------|
| Private VIF | VGW (VPC 직접 연결) | 단일 VPC 연결 (레거시 방식) |
| Transit VIF | Direct Connect Gateway → TGW | 멀티 VPC 연결 (권장) |
| Public VIF | AWS Public 엔드포인트 | S3, DynamoDB 등 AWS 퍼블릭 서비스 접근 |

> **권장 구성:** Transit VIF → DXGW → TGW
> 단일 VIF로 모든 계정의 VPC에 접근 가능

---

## Direct Connect Gateway (DXGW) 구성

```
DX Connection
      │
   Transit VIF (BGP 세션)
      │
   Direct Connect Gateway (글로벌, 리전 무관)
      │
   ┌──┴──────────────────────────┐
   │                             │
TGW (ap-northeast-2)      TGW (ap-southeast-1)
(서울 리전 허브)            (DR 리전 허브)
```

### DXGW 설정 주의사항

- DXGW는 글로벌 리소스 (리전 선택 없음)
- DXGW ↔ TGW 연결 시 **허용 CIDR prefix** 명시적 설정 필요
- DXGW 하나에 최대 **20개 TGW** 연결 가능
- 온프레미스 ↔ AWS 간 CIDR 중복 절대 금지

---

## 고가용성 구성 (HA)

### 권장 구성: 이중화 DX + VPN 백업

```
온프레미스
    │
    ├── DX Connection #1 (Primary)  → DX Location A
    │                                      │
    └── DX Connection #2 (Secondary) → DX Location B (가능하면 다른 위치)
                                            │
                                   Direct Connect Gateway
                                            │
                                      TGW (Network Account)
                                            │
                                  ┌─────── VPN (백업) ────────┐
                                  │  (인터넷 경유 IPsec 터널)  │
```

### 가용성 등급별 구성 옵션

| 등급 | 구성 | 설명 |
|------|------|------|
| 기본 | DX 1회선 + VPN 백업 | 비용 절감, DX 장애 시 VPN 자동 전환 |
| 권장 | DX 2회선 (다른 DX Location) + VPN 백업 | DX Location 장애 커버 |
| 최고 | DX 2회선 × 2 CE 라우터 (다른 DX Location) | 단일 장애점 없음 |

---

## BGP 라우팅 설계

DX는 BGP(Border Gateway Protocol)로 온프레미스 ↔ AWS 간 경로를 교환합니다.

### BGP 설정 원칙

```
온프레미스 → AWS 방향:
  - 온프레미스 CIDR을 BGP로 광고
  - 구체적인 prefix를 광고 (요약 경로보다 구체적 경로 우선)

AWS → 온프레미스 방향:
  - TGW에 연결된 VPC CIDR이 DXGW를 통해 온프레미스로 광고됨
  - 허용할 CIDR은 DXGW의 Allowed Prefix 목록에 명시
```

### Active-Standby 구성 (DX 이중화 시)

| 설정 | 방법 |
|------|------|
| Primary 회선 우선 | Primary에서 더 구체적인 prefix 광고 |
| Standby 회선 대기 | AS Path Prepending 또는 Local Preference 조정 |
| 자동 Failover | BGP Hold Timer 설정으로 장애 감지 시간 단축 |

---

## 암호화 (MACsec)

DX 기본 연결은 암호화되지 않습니다. 데이터 기밀성이 필요한 경우:

| 방법 | 설명 | 적용 레이어 |
|------|------|-----------|
| MACsec | Layer 2 암호화, DX 전용 포트에서 지원 | 회선 전체 |
| IPsec over DX | DX 위에 VPN 터널 구성 | Layer 3 |

> 금융, 의료 등 규정 준수 환경에서는 **MACsec 또는 IPsec over DX** 적용 권장

---

## 온프레미스 DNS 연동

Direct Connect 연결 후 온프레미스 ↔ AWS 간 DNS 조회를 위한 설정이 필요합니다.

```
온프레미스 DNS 서버
        │ (DX 경유)
Route53 Resolver Inbound Endpoint  ← AWS 도메인 질의 수신
        │
Route53 Private Hosted Zone

Route53 Resolver Outbound Endpoint  → 온프레미스 도메인 질의 전달
        │ (DX 경유)
온프레미스 DNS 서버
```

### Resolver Endpoint 배치

- **Inbound Endpoint:** Network Account VPC의 Private Subnet
- **Outbound Endpoint:** Network Account VPC의 Private Subnet
- 워크로드 계정에서 TGW를 통해 Network Account의 Resolver로 DNS 요청 전달
- Forwarding Rule은 RAM으로 전체 계정에 공유
